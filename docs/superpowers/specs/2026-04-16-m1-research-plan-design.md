# M1: 調査・準備 タスク計画

## 背景

e-Gov法令API V2を活用したClaude Code Agent Skillを開発し、skills.shマーケットプレイスに公開するプロジェクト。
実装着手前に、利用規約・ライセンス・保守体制の調査とLICENSEファイル作成を完了させる必要がある（M1マイルストン）。

## 対象Issue

| # | タイトル | 依存 |
|---|---------|------|
| #2 | e-Gov法令APIの利用規約・ライセンス調査 | なし |
| #3 | skillsマーケット公開のライセンス選定調査 | #2 |
| #4 | skillsマーケット公開の保守体制調査 | なし |
| #5 | LICENSEファイル作成 | #3 |

## 実行方針

- 依存関係のないタスクは並列エージェントで同時実行
- 調査結果はIssueコメントに投稿し、ユーザー承認後にクローズ
- #5のLICENSEファイルはPRで作成
- Web調査の手段: WebFetch/WebSearchを基本とし、JS描画が必要な場合はPlaywright MCPを使用

## フェーズ構成

### フェーズ0: 環境準備

1. `.gitignore` に `docs/` を追加し、設計ドキュメント等がリポジトリに含まれないようにする
2. 各フェーズの作業開始時に `/use-git-worktrees` でワークツリーを作成し、mainブランチから隔離された環境で作業する

### フェーズ1: 並列調査（#2 + #4）

2つの独立した調査を並列エージェントで同時実行する。

#### エージェント1 — Issue #2: e-Gov法令API利用規約調査

調査対象:

- e-Gov法令APIの公式ページ（laws.e-gov.go.jp）のヘルプ・利用規約
- Swagger UI / Redoc の利用条件記載
- 政府標準利用規約との関係

調査項目:

1. 利用規約の全体内容
2. 商用利用の可否
3. 帰属表示要件（クレジット表記の必要性）
4. API呼び出し頻度の制限・推奨事項
5. データの二次利用に関する制約

成果物: Issue #2にコメント投稿

#### エージェント2 — Issue #4: 保守体制調査

調査対象:

- skills.shマーケットプレイスの既存人気スキルのリポジトリ
- Agent Skills公式ドキュメント・ベストプラクティス

調査項目:

1. バージョニング戦略（SemVer等）
2. 更新頻度の目安（API仕様変更への追従）
3. Issue対応方針
4. CHANGELOG管理の要否
5. 既存の人気スキルの保守体制を参考調査

成果物: Issue #4にコメント投稿

#### ユーザー確認ポイント

両Issueのコメント内容をレビュー → 承認またはフィードバック

### フェーズ2: ライセンス選定調査（#3）

フェーズ1の#2結果を入力として使用する。

#### エージェント3 — Issue #3: ライセンス選定調査

調査対象:

- skills.shマーケットプレイスの既存スキルのライセンス傾向
- Agent Skills仕様でのライセンス推奨事項
- e-Gov API利用規約との整合性（#2の結果を踏まえて）

調査項目:

1. 既存スキルで採用されているライセンスの傾向（Apache 2.0, MIT等）
2. Agent Skills仕様でのライセンスに関する推奨事項
3. e-Gov APIの利用規約との整合性確認
4. ライセンス候補の比較と推薦

判断が必要な場面:

- 商用/無償の方針 → ユーザーに確認
- 著作権表記の方針 → ユーザーに確認
- ビジネス的思惑に関わる選択 → ユーザーに確認

成果物: Issue #3にコメント投稿（推薦理由つき）

#### ユーザー確認ポイント

ライセンス推薦を承認 → ライセンス確定

### フェーズ3: LICENSEファイル作成（#5）

1. 確定ライセンスに基づきLICENSEファイルを作成
2. ブランチ `issue/5-license` を作成
3. PRを作成し、ユーザーにレビュー依頼
4. マージ後、Issue #5をクローズ

## 完了条件

- 全4 Issue（#2, #3, #4, #5）がクローズされている
- LICENSEファイルがmainブランチにマージされている
- M1マイルストンがクローズ可能な状態
