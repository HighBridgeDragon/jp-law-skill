# スキル統合テスト 実行ガイド

このドキュメントは Issue [#14](https://github.com/HighBridgeDragon/jp-law-skill/issues/14)
「スキル統合テスト（Claude Code動作確認）」の実行手順をまとめたものです。
Claude Code 上で `jp-law` スキルを読み込み、実ユースケース 5 件を実行して
動作を検証します。結果は [integration-test-results.md](integration-test-results.md)
に記入してください。

## 目的

`jp-law` スキルが Claude Code 環境で次の挙動を満たすことを実プロンプトで確認する:

- 法令名の通称・略称から `law_id` を解決できる
- 同梱スクリプト（`scripts/*.sh`）を適切に呼び分ける
- 大規模法令で `elm` パラメータを使い不要な全文取得を回避する
- ヒットしないキーワードに対し「該当なし」を明示する

## 前提環境

| 項目 | 推奨値 |
|---|---|
| Claude Code | v1.x 以降（プラグイン経由でスキル読み込み可能なバージョン） |
| OS | Windows 11 / macOS / Linux のいずれか |
| シェル | bash（Windows の場合 Git Bash または WSL） |
| 必須ツール | `curl`、`bash` |
| ネットワーク | e-Gov 法令API V2 (`https://laws.e-gov.go.jp/api/2/`) に到達可能 |

スキル読み込み方法のいずれかを選択:

- プラグイン読み込み: リポジトリルートで `claude --plugin-dir .`
- ローカルスキル配置: `~/.claude/skills/jp-law` 等にディレクトリ配置

## 観測すべきメトリクス

各テストケース実行中に以下を記録してください。記録欄は
[integration-test-results.md](integration-test-results.md) の各 TC セクションに
あります。

- **呼び出されたスクリプト**: `scripts/<name>.sh` のうち実際に呼ばれたもの
- **引数**: `law_id`、`elm` パラメータ、`limit` 等
- **API 呼び出し URL**（curl ログ等から取得可能な場合）
- **レスポンスサイズ**: 概数（例: 約 2KB / 約 50KB）。`elm` 指定の有無で大きく変動
- **トークン消費**: Claude Code の表示するメトリクス（取得可能な場合）

## 結果記入の手順

1. [integration-test-results.md](integration-test-results.md) を開く
2. 「実行環境」表に日時・Claude Code バージョン・OS 等を記入
3. 各 TC の実行後に以下を埋める:
   - 「呼出ログ抜粋」「レスポンス抜粋」（実際の出力からコピー）
   - 「必須条件チェック」「望ましい条件チェック」のチェックボックス
   - 「判定」を `PASS` / `FAIL` に書き換え
   - 「所見」に気づきや改善提案を記述
4. 「サマリー」を集計し直す（PASS 数 / 5）
5. FAIL があれば「発見した問題と起票 Issue」に別 Issue 番号を記録
6. 全 TC 終了後にコミット → Issue #14 に結果コメント → クローズ

## 判定方針

各 TC は次の二段構えで判定します:

- **必須条件**（all must hold）: すべて満たさなければ FAIL
- **望ましい条件**（best effort）: 満たさなくても PASS だが、満たすほど品質が高い

主観的になりやすい部分（TC-4 / TC-5）は必須条件を最小限に抑え、望ましい条件で
品質を測ります。FAIL は別 Issue として起票し改善トラッキングします。

## TC-1: 民法第709条の表示

### 目的

法令通称からのエイリアス解決と、`elm` パラメータによる条文単位取得が機能することを確認する。

### 入力プロンプト

```text
民法第709条を表示して
```

### 期待される動作

1. `references/law-aliases.md` を参照し `民法` → `129AC0000000089` を解決
2. `bash scripts/fetch-law.sh 129AC0000000089 MainProvision-Article_709` を呼出
3. レスポンスから第709条本文を抽出してユーザーに提示

### 判定基準

**必須条件**:

- `references/law-aliases.md` で `民法` → `129AC0000000089` を解決
- `scripts/fetch-law.sh` を `elm` に `MainProvision-Article_709`（または同等の条文単位 elm 指定）を指定して呼出
- 第709条本文（不法行為に基づく損害賠償の条文）を返却

**望ましい条件**:

- 条文番号・条見出しが正確
- 出典 URL（`https://laws.e-gov.go.jp/law/129AC0000000089` 形式）を提示
- elm 指定により全文取得を回避していることがログから読み取れる

### 既知の難しさ・観測ポイント

- `elm` の階層表記は `MainProvision-Article_709` のようにアンダースコア区切り
- 民法は大規模法令のため、誤って `elm` 無しで取得すると数万行のレスポンスになる
- 出力フォーマットは `SKILL.md` 末尾の推奨フォーマットを参考にする

## TC-2: 個人情報保護法の検索

### 目的

法令名検索（`search-laws.sh`）で正式名称が異なる通称から候補を提示できることを確認する。

### 入力プロンプト

```text
個人情報保護に関する法律を検索して
```

### 期待される動作

1. 入力から「個人情報保護」等の名称キーワードを抽出し `search-laws.sh` に渡す
2. `bash scripts/search-laws.sh 個人情報保護` を呼出
3. 候補リスト（複数）を順序付きで提示
4. 各候補に `law_id` を併記

### 判定基準

**必須条件**:

- `scripts/search-laws.sh` で名称マッチを実行
- 候補の `law_id` を提示
- 複数候補がある場合は順序付きで列挙

**望ましい条件**:

- 正式名称「個人情報の保護に関する法律」(`415AC0000000057`) が候補に含まれる
- マイナンバー法など類似法令も併記
- 各候補に簡潔な説明（公布年・略称等）を付与

### 既知の難しさ・観測ポイント

- API レスポンスの `laws[].law_info.law_id` を抽出する必要がある
- 検索結果は最大 `limit` 件（既定 10）。件数が多い場合の絞り込み判断を観測する

## TC-3: 会社法の改正履歴

### 目的

`fetch-revisions.sh` で改正一覧を取得し、新しい順で提示できることを確認する。

### 入力プロンプト

```text
会社法の改正履歴を見せて
```

### 期待される動作

1. `references/law-aliases.md` を参照し `会社法` → `417AC0000000086` を解決
2. `bash scripts/fetch-revisions.sh 417AC0000000086` を呼出
3. レスポンスの `revisions[]` から改正一覧を新しい順で提示

### 判定基準

**必須条件**:

- `references/law-aliases.md` で `会社法` → `417AC0000000086` を解決
- `scripts/fetch-revisions.sh` を会社法 `law_id` で呼出
- 改正一覧を新しい順で返却

**望ましい条件**:

- 各改正に施行日と改正法名を含む
- 件数が多い場合は直近 N 件に絞って提示
- `law_revision_id` で特定時点の本文取得が可能であることを示唆

### 既知の難しさ・観測ポイント

- 会社法は改正回数が多く、レスポンスが大きい
- API は新しい順で返すが、エージェントが並べ替えしないか確認

## TC-4: AIに関する法律の有無

### 目的

ヒットしないキーワードに対して `search-keyword.sh` を活用し、「該当なし」を明示できることを確認する。

### 入力プロンプト

```text
AIに関する法律はあるか
```

### 期待される動作

1. `bash scripts/search-keyword.sh AI`（必要に応じて `人工知能` 等で再試行）を呼出
2. ヒット件数を明示
3. ヒット 0 件の場合は「該当なし」と明言

### 判定基準

**必須条件**:

- `scripts/search-keyword.sh` を実行（`AI` / `人工知能` 等のクエリを試行）
- 試行したクエリと件数を明示
- ヒット 0 件の場合は「該当なし」を明示

**望ましい条件**:

- 複数クエリ（AI / 人工知能 / 機械学習 等）を順次試行
- ヒットした場合は法令名と該当条文の冒頭を抜粋
- ヒットしない場合は関連法令（個人情報保護法等）の参照を提案

### 既知の難しさ・観測ポイント

- 「AI」は短い英字でヒットが多すぎる/少なすぎる両極端の可能性
- 判定は主観的になりやすいので、必須条件は最小限に絞っている
- `search-keyword.sh` は `AND` / `OR` / `NOT` / ワイルドカード対応

## TC-5: トークン節約ガイダンス

### 目的

大規模法令の操作で `elm` 指定なしの全文取得を回避するガイダンスが機能していることを、TC-1〜TC-4 の呼出ログを通じて確認する。

### 入力プロンプト

TC-1〜TC-4 の操作を通じて観測（独立プロンプトなし）。

### 期待される動作

- 民法・会社法等の大規模法令への問い合わせで `fetch-law.sh` を `elm` 指定で呼ぶ
- 章・節単位のまとめ取得（例: `MainProvision-Chapter_3`）が必要なら適切に選択
- 検索 → 特定 → 取得の順序で進行する

### 判定基準

**必須条件**:

- 大規模法令の操作で `elm` 指定なしの全文取得を回避
- `elm` パラメータが条文単位（例: `MainProvision-Article_709`）で指定されている

**望ましい条件**:

- 章・節単位（例: `MainProvision-Chapter_3`）でのまとめ取得が適切に選択される
- 検索 → 特定 → 取得の順序で進行
- 全文取得を実行する場合はユーザーに確認

### 既知の難しさ・観測ポイント

- 単独プロンプトでは検証しづらいため TC-1〜TC-4 の副次的な観測で判断
- ユーザーが明示的に「全文取得して」と指示した場合は除外（必須条件は適用しない）
- `SKILL.md` 「トークン節約ガイダンス」節 (L107〜L114) が判定の根拠

## TC-6: 会社法第3編第2章のまとめ取得

### 目的

`fetch-law.sh` の `elm` パラメータで章単位（`Part_X-Chapter_Y`）を指定したまとめ取得が、実 API で意図通り動作することを確認する。TC-5 の望ましい条件で N/A となっていた章・節単位取得をカバーする。

### 入力プロンプト

```text
会社法の第3編第2章を表示して
```

### 期待される動作

1. `references/law-aliases.md` を参照し `会社法` → `417AC0000000086` を解決
2. `bash scripts/fetch-law.sh 417AC0000000086 MainProvision-Part_3-Chapter_2` を呼出
3. 第3編第2章配下の全条文をまとめて返却（条文単位の連続取得を回避）

### 判定基準

**必須条件**:

- `references/law-aliases.md` で `会社法` → `417AC0000000086` を解決
- `scripts/fetch-law.sh` を `elm` に章単位指定（`MainProvision-Part_3-Chapter_2` 等）で呼出
- レスポンスに該当章配下の複数条文（`Article` ノード）が含まれる
- 法令全文取得は行われない

**望ましい条件**:

- 章タイトル（`ChapterTitle`）が含まれる
- N 件まとめ取得時のレスポンスサイズが N 回の条文単位取得の合計より大幅に小さい
- 出力フォーマットは SKILL.md 末尾の推奨フォーマットに沿う

### 既知の難しさ・観測ポイント

- 章・節構造を持たない法令（民法第709条のような単条文取得）とは異なり、まとめ取得時のレスポンス階層が深くなる
- 派生検証として附則（`SupplProvision[1]`）、項単位（`Paragraph_X`）、枝番号（`Article_398_22`）の elm 表記も今後追加余地がある

## トラブルシューティング

### スキルが認識されない

- `claude --plugin-dir .` を実行したカレントディレクトリにルートの `.claude/` が
  あるか確認
- `jp-law/SKILL.md` のフロントマター（`name`, `description`）が正しいか確認

### スクリプトが失敗する

- `bash scripts/fetch-revisions.sh 129AC0000000089` 等を手動実行して切り分け
- ネットワーク・プロキシは [VALIDATION_EXECUTION.md](VALIDATION_EXECUTION.md) を参照

### 判定に迷う

- 「必須条件」を厳密に当てはめれば判定可能。すべて満たせば PASS
- 望ましい条件は満たさなくても PASS とし、所見欄に改善提案を記載
- それでも判別不能な場合は FAIL とし、別 Issue で議論
